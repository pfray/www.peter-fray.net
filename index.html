/ shell script to start jenkins on mac

#!/bin/sh
NAME=jenkins 
SCRIPT='/usr/bin/java -- -jar /Users/Shared/Jenkins/bin/jenkins.war --httpPort=7070'
RUNAS=jenkins
 
PIDFILE=/var/run/jenkins.pid
LOGFILE=/var/log/jenkins.log
 
start() {
  echo 'Starting service' >&2
  #local CMD="$SCRIPT &> \"$LOGFILE\" & echo \$!"
  /usr/bin/java -jar /Users/Shared/Jenkins/bin/jenkins.war --httpPort=7070 &
  su jenkins -c "$CMD" $RUNAS > "$PIDFILE"
  echo 'Service started' >&2
}
 
stop() {
  echo 'Stopping service' >&2
  kill -15 $(cat "$PIDFILE") && rm -f "$PIDFILE"
  echo 'Service stopped' >&2
}
 
case "$1" in
  start)
    start
    ;;
  stop)
    stop
    ;;
  retart)
    stop
    start
    ;;
  *)
    echo "Usage: $0 {start|stop|uninstall}"
esac

POC Success Story

POC

Apcera Inc.

/ POC Performed at a Apcera Customer

POC background notes:

The First contact with the prospect was on Aug 27 2015. The POC was executed from Nov 2 to Nov 6th 2015. Follow up questions ensued from the POC up until the close. The deal closed in June 2016 for over 1M.

The prospect environment was very difficult and they are an unusual customer.  They provided absolutely ZERO feedback on who our competition was, or how well Apcera was doing against the competition.  They were very tight lipped. Normally, during a POC, a customer will reveal details on who the competition is and how well you are doing against the competition.  This prospect shared nothing. They also did everything by committee.   Basically, if everyone didn't give a thumbs up during a vote then they would hold up the entire process until everyone was satisfied.  This in fact happened many times in the months following the POC.

Preparation

In preparation for the POC I did many things to insure the POC ran 100%.  I knew we were going to use the docker API and because the Apcera development team was rewriting the API I delayed the POC for 6 weeks so I could show a better experience.  This was going to be largely untested code, fresh off the press, I needed a backup plan.  The backup plan was to learn the existing API and test the POC uses cases against it, which required many API calls just to run a single docker container. At the last minute I was delivered the NEW bits for Apcera Enterprise Edition and installed it. I then moved on to testing the new code to make a determination to go with the new or go with the old.  I decided to use the new environment and "risk it".  The development team did a great job on the API and everything went smoothly.  If it didn't we would have been kicked out of the POC.  In addition, I did complete pull testing and documented it.  Lastly, I tested a complete install in a vSphere environment before going onsite.  The PLAN was to know the results of every test the prospect wanted to perform BEFORE going onsite so I would have a story in place for any deficiencies.  

POC Execution

The POC was executed with parallel paths.  The API path started on Day 1 in the Apcera existing POC environment.  In separate POC path I completed a Apcera installation.  Once the local system was up and running API team switched from using the Apcera environment to using their local environment on vSphere.  After the install I moved on to pull testing.  

Interesting POC facts:

1) No less than 32 Apcera employees were involved with the POC in some way or form.  The prospect liked this because that is the way they "sell their software."  I showed a slide in the wrap-up meeting that showed the number "100000" and asked if they knew what this was.  It is binary number 32 to represent the Apcerains who worked on the POC.

2) The prospect had a scoring criteria and they would ONLY give credit if THEY actually tested the items.  One of the test criteria was scaling (a bomb set by one of our competitors).  The prospects Apcera installation was limited on RAM and would only allow for approximately 150 applications instances to run concurrently.  As a result the prospect would not give us credit for the scalability use case.  So I scaled up our POC AWS environment to have almost a TB of RAM and executed 1,000 unique applications each with 2 instances.  In the POC Wrap-up meeting I showed a slide with "1000/2000" and asked them if they knew what this meant.  None of the prospect employees had any idea. I showed them our AWS environment with all those instance running.  Apcera may not have gotten the check box for scalability but I sure did leave then with the impression that Apcera can scale.

3) There were several items on their requirements list that Apcera did not support.  I provided a roadmap with dates for each of them before leaving the POC.
